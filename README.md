# Тестовое задание Crypton Studio: Concurrent Map

Минималистичная реализация потокобезопасной мапы на Go согласно техническому заданию. В качестве "известного события" выбран **2008 год** — год публикации Whitepaper Bitcoin (Satoshi Nakamoto).

### Задача
Реализовать map, защищённый мьютексом, и написать для него тест. От кода **не требуется** production-ready качество.

### Условия теста
* В мапе ключи должны быть числовыми.
* Количество ключей равно году рождения какой-либо известной личности или году известного события.
* В комментарии к коду указать, откуда взят этот год.
* Диапазон ключей: от `1` до выбранного года.
* Мапу **заранее** ключами и значениями заполнять нельзя.
* В тесте нужно запустить ровно **четыре горутины**.

### Алгоритм работы горутин
1. Обращение к значению по ключу. Если значения нет — оно должно быть создано.
2. Увеличение значения на 1.

⚠️ **Важно:** у мапы **не должно быть** отдельной функции инкремента значения.

### Ожидаемый результат
* После завершения работы всех горутин, для **каждого ключа** должно лежать число `3`.
* Горутины **не должны** обращаться к ключам последовательно (ни по возрастанию, ни по убыванию).

### Требования к структуре мапы
В мапу нужно добавить:
* счётчик обращений к ключам
* счётчик добавлений новых ключей

**В конце теста должно выполняться:**
* счётчик обращений = год * 3
* счётчик добавлений = год

</details>

## Архитектурные решения
Для гарантии того, что горутины не обращаются к ключам последовательно и соблюдают точное количество инкрементов, применен следующий подход:
1. Создан срез ключей, где каждый ключ (от 1 до 2008) повторяется ровно 3 раза.
2. Срез перемешан с помощью `rand.Shuffle`.
3. Ключи отправляются в буферизированный канал, из которого их конкурентно читают 4 воркера.

Это решение исключает гонки данных при распределении задач и делает доступ хаотичным, не усложняя при этом логику блокировок внутри самой мапы.

## Как запустить

Запуск тестов с проверкой на состояние гонки (race detector):
```bash
go test -v -race ./...
```
